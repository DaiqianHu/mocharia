# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Mocha Rush — a Papa's-style coffee shop simulator (Vite, with a **real WebGL 3D scene** via Three.js composited under a 2D HTML5 Canvas HUD). The only runtime dependency is `three`; no other framework. The game evolves through wishlists from a young playtester relayed by the user; keep mechanics forgiving and the tone playful. `mocha-rush.html` at the repo root is an obsolete standalone version of the game kept as a backup — never edit it; all live code is under `src/`.

## Commands

- `npm run dev` — Vite dev server
- `npm run build` — production build (also the fastest syntax/import check; there is no lint or test suite)
- `npm run preview` — serve `dist/`

### Verifying changes (headless)

There is no test suite; verify gameplay changes by driving the real game in headless Chrome with `puppeteer-core` (system Chrome at `/Applications/Google Chrome.app/Contents/MacOS/Google Chrome`). Because the scene is now WebGL, headless Chrome needs software-GL flags. Pattern that works (see `test/harness.mjs` for the shared launcher, and `test/verify.mjs`/`test/play.mjs` for full examples):

1. `npm run build`, then `npx vite preview --port 4231` in the background.
2. Launch Chrome with these flags — **do NOT pass `--disable-gpu`, which turns WebGL off**: `--headless=new --use-gl=swiftshader --enable-webgl --ignore-gpu-blocklist --enable-unsafe-swiftshader` (recent Chrome requires `--enable-unsafe-swiftshader` for WebGL over software rendering). Collect `pageerror`/console errors and take screenshots.
3. **Test more than one window size.** The 2D `#game` canvas is full-window and draws its own letterbox bands; the `#game3d` WebGL canvas is CSS-positioned to *exactly* the inner letterbox rect. At a 1.6 aspect ratio there is no letterboxing, so 960×600 / retina / odd-DPR all coincide — **you must also test a wide (e.g. 1440×720) and a tall (e.g. 900×820) viewport** to exercise real letterbox offsets, which is where alignment bugs hide. Before trusting a screenshot, confirm `document.getElementById('game3d').getContext('webgl2')` is non-null and assert `window.R.info.render.calls > 0` after a `play`-state frame — a blank-but-not-erroring canvas is the likely failure, not a thrown error. (`window.R` is the Three renderer; `window.G`/`window.P` are game state/progression.) Note: `render.calls` is 0 outside `play` by design, and a bare `gl.readPixels` on `#game3d` reads zeros after buffer swap — use `page.screenshot`, which composites both canvases, as the visual source of truth.
4. `page.evaluate` fast-forwards state (set `G.day`, `P.xp`, fill `G.active.cup`/`G.active.cannoli`, empty `G.customers` to force day end) instead of waiting out timers. Scene hit-tests (take order, select machine, drop a topping) go through real button rects and the raycaster, so exercise them by clicking real virtual-pixel coords with `page.mouse` and asserting on `G` state.
5. Mind real-time pacing when clicking: customers take ~4–5s to walk in before "Take Order" enables; machine brews run 13–20s.

## Architecture

Fixed 960×600 virtual resolution (`VW`/`VH` in `core/constants.js`), letterboxed + DPR-scaled (`core/canvas.js`). One `requestAnimationFrame` loop in `src/main.js` calls `update(dt)` (`game/state.js`), then `update3d()` (sync 3D objects to game state, `play` only), then `draw()` (`render/index.js`) every frame. There is no DOM UI.

**Two-canvas compositing (WebGL scene + 2D HUD overlay):** `index.html` stacks two canvases — `#game3d` (WebGL, below) and `#game` (2D, on top, `background:transparent`, keeps all pointer events). In `play` the 3D layer draws the scene (lobby/cup/machines/cannoli/customers) and the 2D layer draws **only HUD** (ticket rail, order panel, tab buttons, station-title cards, hint strips, speech bubbles, particles). Critically, `render/index.js`'s `draw()` must **`clearRect` (transparent), not `fillRect` (opaque)** during `play`, or it hides the WebGL layer; on the 2D-only screens (title/dayIntro/summary/shop) it keeps the opaque `#160d0b` letterbox fill and skips `render3d()` entirely. `#game3d` is CSS-positioned (`syncSize` in `render/three.js`) to exactly cover the inner letterbox rect and the camera aspect stays fixed at VW/VH — this avoids all `setViewport`/scissor offset math and the class of DPR/Y-flip alignment bugs that come with it.

**3D engine (`render/three.js`):** one `THREE.Scene`, one narrow-FOV `THREE.PerspectiveCamera` on `+Z` looking at `-Z` at `distance = (VH/2)/tan(fov/2)` so **world units at z=0 equal virtual pixels** — every hand-tuned anchor in `game/layout.js` (`TOP_CUP`, `MACHINES`, `CANNOLI`) is reused directly as a z=0 placement (world y is up, so virtual `(px,py)` → world `(px-VW/2, VH/2-py, z)` via `toWorld()`/`place()`). Four `THREE.Group`s (`orderGroup`/`brewGroup`/`topGroup`/`cannoliGroup`) toggle by `.visible` from `render3d()`, mirroring `render/index.js`'s station dispatch. Lighting is a hemisphere + two directional lights with a `setHolidayLighting()` hook (reads `currentHoliday()`); **no `THREE.ShadowMap`** — a flat blurred-ellipse `shadowDecal()` sits under each figure. Screen-shake jitters the camera from the same `G.shakeX/shakeY` the 2D layer uses. Shared patterns: **scale-from-anchor fill mesh** (`fillMesh`/`fluidGeometry`, geometry anchored at y=0, driven by `mesh.scale.y = frac` — used for drink fill, brew measuring cup, cannoli cream), and **instanced scatter** (`InstancedMesh` for whip blobs/sprinkles, `TubeGeometry` rebuilt on-throttle for drizzle). Gotcha: **`THREE.Object3D.add()` returns the parent, not the child** — never write `parent.add(child).position.set(...)` (it mutates the parent); add children on separate statements.

**Procedural character rig (`render/character.js` + station reconciliation in `stations/order.js`):** an ~17-bone `THREE.SkinnedMesh` biped built **procedurally in code** from merged primitive geometry with **rigid single-bone skin weights** — authored once as a prototype (`characterProto()`), then `SkeletonUtils.clone()`'d per customer and recolored (skin/shirt/pants/shoes/hair/eye) via per-clone cloned materials (the clone shares the geometry + material-array reference, so materials must be cloned before recolor). Animation is **direct per-frame `bone.rotation` assignment** in `poseCharacter(rig, cust, t)` (a port of the 2D `bob`/`walkA`/`stomp` sine formulas) — **no `AnimationMixer`/`AnimationClip`, no Blender, no Mixamo, no downloaded assets**. `updateOrder3D()` reconciles rigs against `G.customers` (clone on appear, remove on depart). Faceless low-poly heads are a deliberate simplification vs. the old gradient-shaded 2D face.

**Scene builders live with their station:** `stations/*.js` owns **both** its 2D HUD draw (`drawXStation`) **and** its 3D scene (`buildX3D` build-once + `updateX3D` per-frame), not split into parallel `*3d.js` files. `render/scene3d.js` is a thin orchestrator (shared floor/wall `stationRoom3D`, `initScene3d()`, `update3d()` visibility dispatch + per-station update + camera-shake + render — `play` only).

**Deliberate circular imports** extend to the 3D layer: `three.js` is a leaf engine module; `stations/*.js` and `character.js` import its helpers, while `scene3d.js` imports the station builders — safe because every cross-reference runs inside a function at runtime, never at module-eval time. Keep that discipline.

**State machine:** `title → dayIntro → play → summary → shop → dayIntro …`, held in the single mutable global `G` (`game/state.js`). During `play`, `G.station` selects one of four stations: `order`, `brew`, `top`, `cannoli` (list in `game/layout.js`).

**Deliberate circular imports (game logic):** many modules import `G` from `state.js` while `state.js` imports their update functions (e.g. `stations/top.js`, `game/buttons.js`). Safe for the same reason — all cross-references happen inside functions at runtime.

**Core flow of a served drink:**
- `game/data.js` — all content is data-driven catalogs (COFFEES, MILKS, CREAMS, DRIZZLES, SPRINKLE_SETS, RANKS, HOLIDAYS, SHOP_ITEMS). Ingredients carry either `rank: n` (unlocked at that rank) or `holiday: id` (unlocked by buying its shop item while that holiday is active). `makeOrder(day, unlocked)` only draws from unlocked items.
- `game/progress.js` — persistent progression `P` (xp, rank, owned shop items, holiday completion) saved to localStorage key `mocha-rush-save-v2`. Unlock filtering, rank-up math, and shop-effect helpers (`patienceCalm`, `brewSpeed`, `owns`) live here.
- `game/ticket.js` — a `Ticket` records what was actually made: `cup.coffee`/`cup.milk` ({type, temp, amt} poured from machines), topping coverage arrays + exact positions (x in −0.5..0.5 of cup width, ported directly to the 3D cup), and cannoli fill state.
- `game/scoring.js` — pure functions comparing ticket vs order, 0–100 per station (`orderScore` = patience, `brewScore`, `topScore`, `cannoliScore` — null when no cannoli ordered). `serveActive()` in state.js aggregates these into stars/tip/XP; `endDay()` averages them into per-station day ratings for the summary.
- Brew machines are global (`G.machines`, geometry in `layout.js` MACHINES), not per-ticket: configure type/temp/amount 1–3, timed via `COFFEE_TIME`/`MILK_TIME`, then pour into the active ticket.

**Interaction model:** `input.js` self-registers pointer listeners and routes taps through `game/buttons.js` first (`BT` registry + `activeButtons()` per state + `refreshButtonState()` per frame — 2D rect hit-testing; buttons are enabled/labeled there, not where they're drawn). If no button is hit, scene taps use hit-testing that stays in virtual coords where objects sit at z≈0 (machine-select via `machineRect`, customer-tap via the front-customer circle, shelf grabs via `shelfHit`/`cannoliShelfHit`) — projection is ~1:1 there, so no raycast is needed. Only the top-station drop-zone `relX` goes through the raycaster (`hitTestScene(px,py,'top')` in `three.js`, using an invisible collider plane whose `worldToLocal(...).x` reproduces `top.js`'s `-0.48..0.48` mapping). Colliders use `material.visible=false` (mesh stays `visible=true`) so they never render but still raycast. Toppings/cannoli use a drag model: pointer-down on a shelf container sets `G.drag`, and `updateTop`/`updateCannoli` emit topping where held over the target. `core/canvas.js` provides `virtualToNDC()`/`worldToVirtual()` for raycasting and for anchoring 2D popText/particles to 3D objects.

**Audio** (`core/audio.js`): all SFX and the procedural background-music loop (lookahead scheduler, two alternating pentatonic phrases) route through one master gain; `toggleMuted()` persists to localStorage `mocha-rush-muted` and drives the corner `BT.mute` button (drawn on every screen from `render/index.js`). `startMusic()` is idempotent and called from the first `pointerdown` (autoplay policy). The context auto-suspends when the tab is hidden.

**Customers** (`game/customer.js`) never leave once queued: at zero patience they stomp and grumble but wait, punishing only the Order-station rating. Patience drain is scaled by owned decor (`patienceCalm()`).

**Rendering:** the live play scene is WebGL (see the 3D-engine notes above). `render/hud.js` = ticket rail + order checklist panel (2D); `render/screens.js` = title/dayIntro/summary/shop/result overlays (2D-only states, still use `drawShopBackdrop`/`drawWood` from `render/scene.js`). `render/scene.js` also still holds `drawStationLabel` (the HUD title card) and the now-unused 2D `drawCup`/`drawStationRoom`/`drawDecors` (dead code from the pre-WebGL renderer — safe to prune).

**Layout constants** (button rects in `buttons.js`, station geometry in `layout.js`) are hand-placed in virtual pixels; the play area is y≈100 (below the ticket rail, `RAIL_H`) to y≈546 (`TABS_Y`), with the side panel at x≥700 (`PANEL_X`) on non-order stations. Check for overlaps visually via screenshots when moving UI.
